GPU Terrain Generation

Systems / Graphics Programming Experiment

  • Languages: C#, HLSL
  • Tech: Unity, Compute Shaders
  • Focus: GPU Parallelism, LOD, Data Structures

Overview

This project explores real-time terrain generation using the Cubical Marching Squares (CMS) algorithm sampled from a dynamically generated Octree. The entire pipeline—data generation, spatial subdivision, and mesh construction—runs in parallel on the GPU.

Problem Statement

Traditional Marching Cubes struggles with sharp feature preservation and does not integrate cleanly with variable level-of-detail systems. Additionally, recursive spatial data structures such as Octrees are inherently difficult to generate efficiently in a parallel GPU context.

The goal of this project was to design a GPU-friendly terrain pipeline that supports adaptive LOD while preserving surface features and remaining performant.

High-Level Architecture

  • Generate Hermite density and normal data using layered 3D noise
  • Construct an Octree entirely on the GPU via compute shaders
  • Store Octree data in a 3D texture for random-access sampling
  • Apply Cubical Marching Squares to generate mesh geometry

Parallel Octree Generation

Because recursive subdivision introduces race conditions in parallel environments, the Octree is generated in reverse: starting at maximum subdivision and progressively un-subdividing nodes.

  • Octree construction implemented entirely on the GPU
  • Custom node metadata generated using parallel merge-style passes
  • Local GPU memory used to minimize synchronization overhead
  • Efficiency decreases as nodes are un-subdivided, a known tradeoff

Cubical Marching Squares

CMS decomposes the 3D meshing problem into six 2D Marching Squares passes per cube, allowing improved feature preservation and compatibility with variable LOD.

  • Preserves sharp surface features better than Marching Cubes
  • Enables integration with adaptive Octree subdivision
  • Transition faces between LODs present the primary complexity

Technical Challenges

  • Resolving transition faces between neighboring Octree nodes of differing subdivision levels
  • Preventing non-manifold geometry during mesh regeneration
  • Balancing GPU parallelism against memory coherence and access patterns

Performance Considerations

  • All major stages execute on the GPU to minimize CPU-GPU transfers
  • Octree data stored in a 3D texture for cache-friendly access
  • Parallel execution maximized during initial subdivision stages
  • Known inefficiencies emerge during late-stage un-subdivision

Results & Limitations

The system successfully generates airtight meshes with variable LOD on initial execution. However, repeated regeneration can introduce holes at LOD transition boundaries, indicating unresolved edge cases in the transition-face logic.

Learnings & Future Work

  • Recursive data structures require non-intuitive approaches on GPU
  • CMS offers significant advantages for LOD-compatible meshing
  • Future work includes planet-scale terrain and streaming
  • Exploration of advanced noise models and voxel-based fluid systems
